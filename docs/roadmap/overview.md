## What's next

### Thread management

Verdant needs a way to list past conversations and start new ones explicitly. Currently `thread_id` is generated by the client — there is no server-side record of what threads exist for a user.

**Design decisions:**
- Cambium owns thread ID generation — `POST /api/v1/threads` returns a server-assigned ID before any chat begins. The client never picks IDs.
- Thread IDs are **botanical three-word names** (`silver-fern-cascade`, `ancient-lotus-dawn`) generated from three themed word lists (31 descriptors × 41 plants × 36 phenomena ≈ 45,700 unique combinations). On collision, retry once.
- Actual message content lives in the LangGraph PostgresSaver checkpointer. The `Thread` model stores only metadata (title, preview, timestamps) — no message duplication.
- Interrupted streams: the LangGraph checkpoint always contains the complete AI response (saved after `llm_call` node, before streaming to client). On reconnect, `GET /api/v1/threads/{id}/messages` returns the full history from the checkpoint — no Redis or separate cache needed.

**New endpoints:**
```
POST   /api/v1/threads              — generate botanical ID, register with Rhizome, return thread_id
GET    /api/v1/threads              — list user's conversations (sorted by last_active_at)
GET    /api/v1/threads/{id}         — thread metadata + last message preview
GET    /api/v1/threads/{id}/messages — full message history from LangGraph checkpoint
DELETE /api/v1/threads/{id}         — delete metadata + clear LangGraph checkpoint
```

**Requires:**
1. **Rhizome (narcissus branch)** — `Thread` model, `session_context_intake` auto-updates `last_active_at` + `last_message_preview`, thread data endpoints in the data router
2. **Cambium (new branch after narcissus merges)** — `internal/api/threadnames.go` with botanical word lists + generator, `POST /api/v1/threads` handler, proxy routes for remaining thread endpoints

### Rate limiting

Auth endpoints (`/auth/register`, `/auth/login`) need brute-force protection. Options:
- `golang.org/x/time/rate` (in-process, resets on restart)
- Redis-backed rate limiter (survives restarts, works across multiple Cambium instances)

Decision needed before production deployment.

### Multi-tenancy in Rhizome

Currently `user_id == 1` is hardcoded as a fallback in Rhizome's CLI mode. When Cambium provides a real `user_id`, it flows through correctly — but the Rhizome tools need a full audit to confirm every query is properly scoped. This becomes critical before the first multi-user deployment.

### CAMBIUM_ENCRYPTION_KEY rotation

If the master encryption key ever needs rotating, all stored provider keys must be re-encrypted atomically. A rotation script is needed before storing real user keys in production. Pattern:
1. Decrypt all keys with old key
2. Re-encrypt with new key
3. Swap env var + redeploy atomically

### gRPC migration

Switch Cambium→Rhizome transport to gRPC when Verdant needs token streaming with bidirectional flow (e.g. user sends mid-generation interrupts). SSE handles the current unidirectional streaming well; gRPC becomes worth the overhead only when the protocol changes.
