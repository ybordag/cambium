## What's next

### Thread management

Verdant needs a way to list past conversations and start new ones explicitly. Currently `thread_id` is generated by the client and passed as a query parameter — there's no server-side record of what threads exist.

Requires:
1. **Rhizome (narcissus branch)** — thread endpoints in the data router: `POST /internal/data/threads`, `GET /internal/data/threads`, `GET /internal/data/threads/{id}`
2. **Cambium** — proxy routes: `POST /api/v1/threads`, `GET /api/v1/threads`, `GET /api/v1/threads/{id}`

### Rate limiting

Auth endpoints (`/auth/register`, `/auth/login`) need brute-force protection. Options:
- `golang.org/x/time/rate` (in-process, resets on restart)
- Redis-backed rate limiter (survives restarts, works across multiple Cambium instances)

Decision needed before production deployment.

### Multi-tenancy in Rhizome

Currently `user_id == 1` is hardcoded as a fallback in Rhizome's CLI mode. When Cambium provides a real `user_id`, it flows through correctly — but the Rhizome tools need a full audit to confirm every query is properly scoped. This becomes critical before the first multi-user deployment.

### CAMBIUM_ENCRYPTION_KEY rotation

If the master encryption key ever needs rotating, all stored provider keys must be re-encrypted atomically. A rotation script is needed before storing real user keys in production. Pattern:
1. Decrypt all keys with old key
2. Re-encrypt with new key
3. Swap env var + redeploy atomically

### gRPC migration

Switch Cambium→Rhizome transport to gRPC when Verdant needs token streaming with bidirectional flow (e.g. user sends mid-generation interrupts). SSE handles the current unidirectional streaming well; gRPC becomes worth the overhead only when the protocol changes.
